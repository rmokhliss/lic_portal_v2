2. Interactions entre layers

2.1 Schéma

| REST + JWT (Authorization header)

| payloads validés via zod (workspace shared)

| trace_id propagé en header traceparent

v

Backend (NestJS, hexagonal)

+-----------------------------------------------------+

| domain <- application <- ports                              |

|                               ^                             |

|                               |                             |

|                            adapters                         |

|                   +------+------+                           |

|                   v           v      v                      |

|                   Drizzle ioredis nats.js                   |

+-----------+----------------+----------------+-------+

            |                |                        |

   transactions     cache, locks                      events

            v                v                        v

   PostgreSQL                Redis     NATS JetStream

   (source vérité)           (éphémčre) (persistant)

2.2 Frontend  Backend

    · Communication exclusivement via REST sur HTTPS.
    · JWT envoyé en header Authorization: Bearer <token>.
    · Validation token côté backend via NestJS Guards (@s2m/core-auth).
    · Payloads validés des deux côtés avec le męme schéma zod (shared package).
    · trace_id OpenTelemetry propagé en header traceparent (généré côté client si absent).
    · Backend adopte le trace_id comme racine pour corréler logs + SQL + NATS.

2.3 Backend  Datastores

    · PostgreSQL : opérations d'état durable. Transactions ACID pour opérations critiques.
    · Redis : éphémčre uniquement (verrouillage, idempotence, cache, vélocité).
    · NATS : publication d'événements. Consommateurs indépendants.
    · Le backend ne lit jamais Redis ou NATS comme source de vérité. Seul PostgreSQL fait

         foi.

2.4 Datastores entre eux

    · Aucune interaction directe entre datastores.
    · Le backend orchestre toutes les coordinations.

3.3 Templates fournis

Tout nouveau projet S2M doit copier dans son docs/templates/ les fichiers suivants, fournis
par le Référentiel :

    · adr-template.md : format Michael Nygard standardisé pour Architecture Decision
         Records.

    · project-context-template.md : structure type du PROJECT_CONTEXT.md (voir
         §4.21).

Template ADR

    # NNNN -- Titre court de la décision

    ## Status
    Accepted | Proposed | Superseded by NNNN | Deprecated -- <Mois Année>

    ## Context
    Décrit le problčme, les contraintes, les options évaluées (2 ŕ 4 options
    typiquement). Chaque option avec ses pros/cons concrets. Pas de blabla
    théorique : référencer code, sprint, contrainte business existante.

    ## Decision
    La décision retenue, énoncée clairement et bričvement (1-3 paragraphes max).
    Inclut les points d'implémentation concrets : tables, composants,
    bibliothčques choisies.

    ## Consequences

    **Bonnes**
    - Liste des bénéfices concrets (1 ligne par item).

    **Mauvaises**
    - Liste des coűts/limitations acceptés (1 ligne par item).

    **Neutres**
    - Conséquences ni positives ni négatives mais ŕ connaître.

    · Format ADR strictement respecté. Numérotation NNNN séquentielle (0001, 0002, ...). Un
         seul ADR par fichier. Un ADR jamais modifié rétroactivement : si la décision change,
         créer un nouvel ADR avec status "Supersedes NNNN" + ancien passe en "Superseded
         by".

4. Rčgles pour Claude Code

4.1 Réflexe avant codage

    · Lire CLAUDE.md racine puis CLAUDE.md du workspace concerné.
    · Lire PROJECT_CONTEXT.md pour l'état actuel du projet.
    · Scanner le module voisin pour patterns en place.
    · Chercher dans s2m-core et s2m-monetique avant d'écrire (annexe A).
    · Si aucun pattern existant ne couvre le besoin, demander avant d'introduire un nouveau

         pattern.

4.2 Rčgles absolues

Classification des rčgles : Critique, Universel, Contextuel

 Trois niveaux d'applicabilité, ŕ interpréter avant tout audit conformité :
·  Critique : rčgle inviolable, écart impossible (no any, no console.log, no SQL

·  string brut, audit obligatoire, pas de crypto custom). Casse le build/CI.
   Universel : rčgle obligatoire dans tout projet S2M (hexagonal, codes erreur

   S2M-<DOMAIN>-<NUM>, conventions de nommage, dates UTC, EN+FR par

   défaut).
·  Contextuel : rčgle dépendante du périmčtre projet. Obligatoire si critčre rempli,

   sinon ADR justificatif requis. Exemples : multi-tenant + RLS (si >1 client distinct),

   decimal.js (si manipulation de monnaie), Redis (si besoin pub/sub ou idempotence

   inter-services), NATS (si bus inter-services), JWT (si API publique avec clients tiers).

Les rčgles ci-dessous sont par défaut Critique ou Universel sauf mention contraire. Les

briques Contextuel sont identifiées dans §1 par le rôle qu'elles jouent.

 number natif sur montants Contextuel (si manipulation de monnaie)

Pour toute valeur monétaire ou financičre, utiliser exclusivement decimal.js (via
@s2m/core-decimal). Recherche `: number` dans domain/ doit retourner zéro résultat sur
les modčles financiers. ADR requis si projet sans monétique (volumes entiers uniquement).

 Cryptographie custom Critique

Aucune implémentation maison de HMAC, AES, RSA, signatures. Utiliser exclusivement
@s2m/core-crypto.

 Audit sur mutations d'état Critique

Toute modification d'état critique (compte, transaction, configuration, utilisateur) appelle
AuditRecorder. Inclut succčs ET échecs critiques (champ outcome=SUCCESS|FAILURE).

 Erreurs typées Critique

Pas de `new Error("...")` ni `throw "string"`. Utiliser @s2m/core-errors. Code S2M-
<DOMAIN>-<NUM>.

 Architecture hexagonale stricte Universel

· domain/ : aucune dépendance hors TypeScript et decimal.js

· application/ : dépend de domain et ports uniquement

    · adapters/ : implémentent les ports, peuvent dépendre de tout sauf d'autres adapters
    · Vérifié automatiquement par dependency-cruiser ou eslint-plugin-boundaries.

 console.log en code applicatif Critique

Utiliser le logger pino injecté via @s2m/core-observability. console.log toléré uniquement en
débogage local temporaire.

 Sérialisation montants Contextuel (si monnaie)

· Tout montant traverse le réseau en string, jamais en number.

· Schéma zod partagé valide deux champs liés : amount (string) + currency (ISO
    4217).

· Décimales selon devise : JPY=0, EUR/USD=2, MAD/TND=3, KWD/BHD=3, BTC=8.
· Validator dédié : amountForCurrency (@s2m/core-validate).

    · Conversion en Decimal ŕ la frontičre (controller backend ou couche API frontend)
         avant calcul.

 JWT : access en mémoire + refresh en cookie Contextuel (variante A)

· Access token : 15 min, stocké en mémoire JS uniquement (jamais
    localStorage/sessionStorage).

· Access transmis via header Authorization: Bearer <token>.

· Refresh token : 7 jours, cookie HttpOnly Secure SameSite=Strict.

· Endpoint /auth/refresh pour renouvellement.

    · Révocation serveur via Redis (jti blacklist).
    · Identité utilisateur : endpoint /me, jamais parser le JWT côté client.

 Multi-tenant Contextuel (si >1 client distinct)

· Toute table métier : colonne tenant_id NOT NULL avec FK et index.

· Row Level Security PostgreSQL activée. Policy USING (tenant_id =
    current_setting('app.tenant_id')::uuid).

· Backend SET app.tenant_id en début de transaction depuis le contexte JWT.

· Garde applicatif : @s2m/core-tenant + decorator @CurrentTenant() injecté dans
    use-cases.

· Requęte sans tenant -> erreur S2M-AUTH-002 avant DB.

4.6 Qualité du code

    · Une fonction = une responsabilité. >30 lignes = signal de scission.
    · Noms explicites long > noms courts cryptiques.
    · Commentaires : pourquoi, jamais quoi. Uniquement pour choix non triviaux.
    · Early returns plutôt que grandes imbrications if/else.
    · const par défaut, immutabilité préférée.
    · Pas d'héritage profond, pas de proxies, pas d'injection magique.
    · Tests obligatoires : unitaires sur domain/, intégration sur application/. Couverture 80%.

4.7 Discipline IA

    · str_replace pour modifications ciblées, pas de réécriture totale.
    · view_range pour lecture partielle des longs fichiers.
    · Pas de ré-énoncé du contenu fourni : référencer par chemin et ligne.
    · Réponses concises, pas de préambule ni conclusion verbeuse.

   Pattern Stop & Validate sur tâches multi-étapes

   Pour toute tâche en plusieurs étapes, l'agent identifie en début de tâche les "stops" ŕ
   appliquer :

        · Stop court (showing) : étapes plombičres sans choix architectural (pnpm install,
             création dossiers vides, lint final, vérifications). Affichage du résultat puis attente
             d'un "ok continue".

        · Stop validation (gating) : étapes contenant un choix de design (tokens DS,
             structure de layout, pattern de composant, signature d'API, schéma de table).
             Affichage code + rendu si applicable. ATTENTE EXPLICITE de l'accord humain
             avant de continuer.

        · Une étape avec choix architectural sans Stop validation = violation de discipline IA.

   Invention et duplication
        · Utiliser strictement la nomenclature du référentiel et du code existant.
        · Pas de synonymes inventés (compte  wallet  account record).
        · Pas de nouveau composant, module ou concept sans validation explicite.
        · Pas de duplication d'un libellé sous deux noms différents.
        · Pas de schéma décoratif. Chaque visuel = réalité technique vérifiable dans le code.
        · En cas de doute : [ŕ valider] entre crochets, jamais d'approximation.

4.13.2 Port et adapter

   Pourquoi abstract class plutôt que interface

        · Avec NestJS : abstract class fournit un token concret pour le DI container
             (@Inject(AccountRepository)). Une interface TypeScript disparaît ŕ la
             compilation, le DI ne peut pas la résoudre.

        · Avec Next.js full-stack ou DI manuelle : abstract class et interface sont
             fonctionnellement équivalentes pour le typage. abstract class reste recommandée
             pour cohérence cross-projets et compatibilité future avec NestJS.

        · Pattern de mock en tests : extends de l'abstract class avec implémentations stub
             ou Vitest fn(). Voir §4.13.5.

4.13.6 Server Action Next.js (variante full-stack)

Pour les projets en variante Next.js full-stack (voir §4.12), la Server Action remplace
fonctionnellement le Controller HTTP NestJS du §4.13.4. Le pattern hexagonal et les use-cases
restent inchangés.

   Rčgles d'une Server Action

        · Nommage : <verb><Entity>Action (ex: createClientAction, updateLicenceAction).
        · Premičre ligne : "use server".
        · Vérification du rôle via requireRole() AVANT toute opération.
        · Validation Zod stricte du payload (parse, jamais safeParse silencieux).
        · Appel exclusif d'un use-case dans application/. AUCUNE logique métier dans la

             Server Action.
        · revalidatePath ou revalidateTag aprčs mutation pour invalider les caches Server

             Components.
        · Retour DTO via toDTO() du mapper, jamais l'entité de domaine brute.
        · Audit dans la męme transaction que la mutation (via le use-case).

// app/src/app/(dashboard)/clients/_actions.ts
"use server";

import { revalidatePath } from "next/cache";
import { CreateClientSchema, type ClientDTO } from
"@/shared/schemas/client.schema";
import { requireRole } from "@/server/infrastructure/auth";
import { createClientUseCase } from "@/server/modules/client/client.module";
import { toDTO } from "@/server/modules/client/adapters/postgres/client.mapper";

export async function createClientAction(input: unknown): Promise<ClientDTO> {
   // 1. Vérification rôle (côté serveur, jamais uniquement UI)
   await requireRole(["ADMIN", "SADMIN"]);

   // 2. Validation Zod stricte
   const parsed = CreateClientSchema.parse(input);

   // 3. Appel use-case (transaction + audit gérés ŕ l'intérieur)
   const client = await createClientUseCase.execute(parsed);

   // 4. Invalidation cache Server Components
   revalidatePath("/clients");

   // 5. Retour DTO sérialisable client
   return toDTO(client);
}

4.16 Sécurité

   Rate limiting
